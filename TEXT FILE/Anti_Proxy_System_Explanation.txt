SMARTATTEND - ANTI-PROXY SYSTEM EXPLANATION
========================================================================
Date of Update: June 7, 2026

WHAT WE HAVE DONE TODAY
-----------------------
Today, we successfully resolved a major "False Positive" bug in the Lab Attendance system. Previously, the system used a "Hardware Fingerprint" to block proxy attendance. The problem was that if two innocent students owned the exact same model of phone, the system thought they were the same person and falsely blocked the second student.

To fix this, we completely removed the Hardware Fingerprint tracking and replaced it with a purely "LocalStorage UUID (DeviceID)" system. 

We prioritized the student experience: zero innocent students should ever face an issue. We removed background flagging completely. The system now perfectly blocks normal proxies (same browser session) and relies on the professor's simple visual headcount to catch any extreme edge cases (like clearing browser cache).

------------------------------------------------------------------------
WHAT IS A DEVICE ID (LOCALSTORAGE UUID)?
------------------------------------------------------------------------
When a student logs into the SmartAttend portal for the first time, our code generates a completely random, mathematically unique text string called a UUID (Universal Unique Identifier) — for example: `id_4f8g9k_1701234567`.
This string is saved permanently in the browser's "Local Storage". It acts like a digital ID card that is unique to that exact browser session, on that exact phone.

------------------------------------------------------------------------
WHAT IS THE USE OF THE "DEVICE ID" COLUMN?
------------------------------------------------------------------------
Every time a student submits a 6-digit lab code, their browser sends its saved DeviceID to the backend. The backend records it in the `Lab_Access_Logs` sheet.
Rule: If Student B tries to mark attendance, and their DeviceID perfectly matches a DeviceID already recorded for Student A for that same session, the system INSTANTLY BLOCKS it. This prevents 99% of normal proxy attempts (like handing your phone to your friend to mark their attendance).

------------------------------------------------------------------------
PROPER EXAMPLES OF HOW THE SYSTEM REACTS
------------------------------------------------------------------------

SCENARIO 1: The Normal Proxy Attempt
- Student A marks attendance on their phone. (DeviceID: xyz)
- Student A passes their phone to Student B.
- Student B types their own ID and the 6-digit code.
- System Action: BLOCKED. The backend sees DeviceID "xyz" was already used by Student A.

SCENARIO 2: Two Innocent Students with Identical Phones
- Student A marks attendance on their iPhone. (DeviceID: abc)
- Student B marks attendance on their own iPhone (same exact model). (DeviceID: def)
- System Action: ALLOWED. Because the DeviceIDs are different, they are perfectly allowed with zero flags or errors.

SCENARIO 3: The Clever Cheater (Incognito/Cleared Cache)
- Student A marks attendance. (DeviceID: ghi)
- Student A clears their browser cache or opens an Incognito tab to proxy for Student B.
- The browser generates a new DeviceID.
- Student B's attendance is submitted. (DeviceID: jkl)
- System Action: ALLOWED. The professor handles this edge case by doing a quick 30-second visual headcount in the lab. If the headcount is 39 but the app says 40, the professor knows someone cheated.
